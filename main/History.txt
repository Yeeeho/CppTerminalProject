- 생성자 주입을 통해 상속되지 않은 클래스 사이에 인스턴스 이동
- 부모 클래스는 자식 클래스에서 무엇을 하는지 알 수 없음. 
- 예를 들어 자식 클래스의 자료형을 담는 벡터를 부모 클래스에서 선언하면 오류가 발생함.
- 필요하다면 다른 클래스에서 다뤄야 함.

- 부모클래스 객체를 자식클래스 객체로 형변환하는 것을 다운캐스팅이라고 함.
- 남발하면 의존성 문제가 생길 수 있으므로 주의해야 함.

객체 슬라이싱. 아이템 타입과 적 타입의 다운캐스팅 실패에 관하여
- 엔티티 타입의 복사 생성자로 에너미 타입의 인스턴스를 복제하면
- 컴파일러가 부모 타입의 메모리만 할당하게 되고, 오버라이딩 정보가 사라지게 된다.

- Item* i 라는 객체가 있고, Boot* b가 있다.
- *i 즉, 실제 오브젝트에 하위 클래스의 값 *b을 대입시키면 메모리 크키가 달라서 슬라이싱이 발생한다.
- 즉, 오고곡 이렇게 큰건 안들어가욧 과 같다고 보면 된다.

- 하지만 Item* i 같이 포인터에다가 하위 클래스의 포인터를 대입시키면 주소만 복사하는 거라
- 슬라이싱은 발생하지 않는다. (예시: Item* i = Boot* b)
- 실제로 포인터들은 일정한 메모리를 할당받는다.

- 클론 함수를 만들어서 원래 무슨 타입이었는지 기억하게 만드는게 좋다.. 아니 필수다..
- 가상함수를 오버라이딩 해서 자기 자신을 복사 생성자로 복제해서 리턴하는 함수다.
- 이걸 프로토타입 패턴이라고 부른다.

포인터 문제..
- 빈 포인터를 선언할 경우 명시적으로 nullptr로 초기화해야 함. 그렇지 않으면 쓰레기 값을 가리킬수 있음.
- nullptr의 메모리에 접근하려고 하면 세그멘테이션 폴트가 발생함. 주의할 것.

포인터의 포인터 문제..
- 포인터는 대입 연산자로 복사할 경우 실제 메모리가 할당되지 않고 그 포인터를 가리키기만 함.
- 그 포인터에 nullptr을 대입하면 해당 포인터가 null이 되는거지 원래 그친구가 가리키던 포인터는 영향이 없음..

- 클래스가 포인터 변수를 멤버로 가지고 있을 경우 복사 생성자에서 메모리를 할당해줘야 함.

포인터를 레퍼런스로 넘겨주고 싶었는데.. ui 클래스에서 item 타입의 인스턴스를 받을때..
- item 타입 인스턴스를 매개변수로 받는 f 함수가 있다.
- item 타입 인스턴스를 생성자로 주입받아서 쓰는 c 클래스도 있다.
- c에서 f로 인스턴스를 넘겼을 때, 함수 내부의 item은 클래스의 item의 주소를 복사한 포인터다.
- 따라서 f에서 item = nullptr을 해도 클래스 내부의 item은 nullptr이 아니다.. 

- 그래서 참조에 대한 포인터(Class&*)가 가능한가? 안된다. 언어에서 이걸 막더라.
- 참조는 실제 객체의 또다른 이름일 뿐, 어떤 메모리 주소를 갖고 있지 않기 때문이다.
- 따라서 포인터가 실제 메모리가 없는 레퍼런스를 가리킬 수는 없다.
- '별명의 주소'라고 하면 이상하지 않은가?

- 그렇다면 포인터에 대한 참조(Class*&)는 가능한가? 이거는 된다.
- 포인터도 메모리에 어떤 객체의 주소값을 가지고 있어서 실제 메모리가 존재하기 때문이다.
- '주소의 별명'이라고 하면 말이 된다.


- 벡터 컨테이너, std::string 자료형은 포인터들을 담고 있다고 해도 복사될 경우 자동으로 메모리를 할당함.
- 스마트 포인터도 마찬가지. 

포인터 함수 공변환 문제..
- 포인터 함수는 공변환이 안된다. 불변성이 있기 때문. 따라서 포인터 함수의 타입을 알아내려면 직접 형변환을 해줘야 한다.
- 아니면 템플릿을 쓰거나 래퍼 함수를 쓰면 된다.

- 레퍼런스는 변수의 별명이라고 생각할 수 있음.
- 포인터가 특정 메모리 주소를 가리키기만 한다면 레퍼런스는 해당 메모리에 또다른 이름으로 등록되어 있음.
- 따라서 레퍼런스는 초기화할 때 무언가를 대입해주어야 함.

shared_prt(공유 포인터)
- 기본 포인터를 복사하면 복사한 포인터가 기본 포인터를 가리키는 구조가 된다.
- 하지만 shared_ptr 은 복사될 때 기본 포인터가 가리키턴 오브젝트를 가리키게 된다.
- 공유 포인터는 내부적으로 카운터를 가지고 있어서 특정 객체를 가리키는 공유 포인터 개수를 알 수 있다.
- 어떤 오브젝트를 가리키는 공유 포인터가 모두 사라지면 포인터가 가리키던 객체도 파괴된다. 

- 멤버 함수 배열 -> 함수 테이블
- 멤버 함수 포인터를 다른 클래스에서 읽기 위해선 &클래스::함수 형식으로 불러와야 한다. 컴파일러가 함수 호출로 착각할수 있다더라.
